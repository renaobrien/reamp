# Reamp Technical Spec

Version 1.0 | July 2026 | Companion to reamp-prd.md

## 1. Platform reality (read this first, it drives everything)

### Spotify (state as of July 2026)

- **Development Mode** (what Reamp will be): 1 client ID per developer, max 5 authorized users on an allowlist, app owner must hold an active Premium subscription, reduced endpoint set. These rules took effect Feb 11, 2026 for new apps and Mar 9, 2026 for existing ones.
- **audio-features and audio-analysis endpoints are dead** for all apps created after Nov 27, 2024. 403, no replacement, no waitlist. The visualizer gets nothing from Spotify. Do not build anything expecting beat/tempo/segment data from Spotify.
- **Web Playback SDK is alive**: creates a local Spotify Connect device in a Chromium context, streams encrypted audio (EME/Widevine), requires Premium, non-commercial use without written approval. This is Reamp's Spotify playback path.
- **Player endpoints survive in dev mode**: transfer playback, play/pause/next/prev/seek, shuffle, repeat, volume, currently-playing, get playlist, get playlist items. Some catalog endpoints (several-albums, several-artists, new-releases, markets, artist-top-tracks) were removed for dev mode in Feb 2026. Playlist creation moved to `POST /me/playlists` (current user only). Verify the exact surviving list against the Feb 2026 changelog and March 2026 reversion notes during implementation; do not trust older SDK wrappers or training data.
- **Auth**: Authorization Code with PKCE. Implicit grant is deprecated. Loopback redirect URIs must use `127.0.0.1`, not `localhost`.
- **Consequence**: BYO client ID. Settings pane walks the user through creating their own app at developer.spotify.com and pasting client ID + redirect URI. Reamp never ships a shared client ID.

### Apple Music

- **MusicKit JS v3** in the Chromium renderer: full-track DRM'd playback for subscribers, library/playlist/catalog access, transport control. Requires an Apple Developer Program membership ($99/yr) to mint a MusicKit developer token (ES256 JWT from a private key, 6-month max expiry, generated by a tiny local script, no server needed).
- Audio is EME-protected. No Web Audio tap on the playback element. Same loopback answer as Spotify.
- Apple Music API still exposes **tempo, key, timeSignature** on many catalog tracks. Use for beat-synced preset transitions (P1), never as the primary vis signal.
- Native Swift MusicKit exists on macOS and visionOS. Phase 3 uses it directly.

### The DRM wall and the loopback answer

Neither service will ever hand over PCM. The visualizer instead captures **system audio output**:

- **macOS (P0)**: ScreenCaptureKit (`SCStream` with audio, macOS 13+). Requires one-time user permission. Implemented as a small Swift sidecar binary that streams Float32 PCM frames over stdout/local socket to the shell process. Zero third-party installs. Fallback documented for users who prefer BlackHole + multi-output device.
- **Windows (P1)**: WASAPI loopback via native module. No permission prompt. Straightforward.
- **visionOS (Phase 3)**: no loopback exists. Apple Music visuals there run on native metadata + animation; this is a known and accepted downgrade.
- **Rule**: PCM lives in a ring buffer, feeds FFT, is never written to disk, never leaves the process. Analysis of one's own audio output for real-time visuals is the defensible posture; keep it strictly that.
- Capture is output-device-wide, so non-music system sounds will hit the analyzer. Acceptable for v1; optional per-app capture via SCK content filters later.

## 2. Architecture

```
┌────────────────────────────────────────────────────────┐
│ Electron shell (macOS)                                 │
│                                                        │
│  Renderer (Chromium)                                   │
│  ├─ Webamp core (UI, .wsz skin engine, windows)        │
│  ├─ Source Adapter layer                               │
│  │   ├─ SpotifyAdapter (Web Playback SDK + Web API)    │
│  │   └─ AppleMusicAdapter (MusicKit JS v3)             │
│  ├─ Vis Engine                                         │
│  │   ├─ Classic vis (canvas spectrum/oscilloscope)     │
│  │   ├─ Butterchurn (Milkdrop, WebGL)                  │
│  │   └─ Modern scenes (WebGPU, P1)                     │
│  └─ Settings / onboarding UI                           │
│                                                        │
│  Main process (Node)                                   │
│  ├─ OAuth loopback server (127.0.0.1, PKCE)            │
│  ├─ Keychain token storage (safeStorage)               │
│  ├─ Sidecar manager ──► capture-sidecar (Swift, SCK)   │
│  └─ Window manager (frameless, snapping, always-top)   │
└────────────────────────────────────────────────────────┘
```

### Shell choice: Electron over Tauri

- Webamp and Butterchurn are mature in Chromium; Spotify Web Playback SDK requires Widevine, which Electron supports via castLabs' Electron for Content Security (ECS) build. Tauri's WebView (WKWebView on macOS) has unreliable Widevine support, which kills the Spotify playback path.
- **Flag: Widevine in Electron requires the castLabs ECS fork and VMP signing for production.** Prototype with ECS from day one so the DRM path is proven in week 1, before any UI investment. If ECS proves painful, fallback architecture: run the Web Playback SDK in the user's default browser as a hidden Connect device page, with Reamp controlling it via the Connect API. Uglier, fully viable, zero DRM burden.

### Source Adapter interface (the contract everything hangs on)

```typescript
interface SourceAdapter {
  id: 'spotify' | 'apple-music';
  auth(): Promise<AuthState>;
  disconnect(): Promise<void>;
  // transport
  play(uri?: string): Promise<void>;
  pause(): Promise<void>;
  next(): Promise<void>;
  previous(): Promise<void>;
  seek(ms: number): Promise<void>;
  setVolume(pct: number): Promise<void>;
  setShuffle(on: boolean): Promise<void>;
  setRepeat(mode: 'off' | 'track' | 'context'): Promise<void>;
  // state
  onPlayerState(cb: (s: PlayerState) => void): Unsubscribe;
  // browse
  getPlaylists(): Promise<PlaylistSummary[]>;
  getPlaylistTracks(id: string): Promise<Track[]>;
  search?(q: string): Promise<Track[]>;        // optional, endpoint availability varies
  getTrackFeatures?(id: string): Promise<{ tempo?: number; key?: string }>; // Apple only
}

interface PlayerState {
  track: { title: string; artist: string; album: string; durationMs: number; artUrl?: string };
  positionMs: number;
  playing: boolean;
  shuffle: boolean;
  repeat: 'off' | 'track' | 'context';
  volume: number;
  bitrateLabel?: string; // cosmetic, for the Winamp kHz/kbps display
}
```

Webamp's media layer is swapped to delegate to the active adapter. Webamp already exposes hooks for custom media; where it assumes Web Audio ownership (EQ, balance), stub with visual-only behavior for streaming sources.

### Vis pipeline

```
SCK sidecar ─► PCM ring buffer (main) ─► SharedArrayBuffer ─► renderer
  renderer: AnalyserNode-equivalent FFT (or meyda/custom radix-2)
  ├─► classic vis canvas (75-band bars, oscilloscope)
  └─► Butterchurn.render(audioLevels) per frame
```

- Butterchurn accepts an audio node or raw time/frequency arrays; feed it the shared buffer directly.
- Target: 1024-sample FFT at 48kHz, ~46ms frames, smoothed; <100ms end-to-end.

## 4. Auth and secrets

- **Spotify**: PKCE flow. Main process opens system browser to authorize URL, catches redirect on `http://127.0.0.1:<port>/callback`, exchanges code, stores refresh token via Electron `safeStorage` (Keychain-backed). Client ID entered by user in settings; validate with a ping to `/me`.
- **Apple**: developer token generated offline by `scripts/gen-apple-token.ts` from the user's `.p8` key (never committed); user token obtained via MusicKit's `authorize()` in the renderer. Both stored via safeStorage.
- No secrets in the repo, no proxy server, everything local.

## 5. Skins

- Webamp handles `.wsz` (zip of BMPs + config) natively, including cursors, viscolor.txt (which colors the classic vis!), and region maps.
- Bundle one CC-licensed community skin as default. Do not bundle Winamp base skin (Llama Group IP). Link users to the Winamp Skin Museum (skins.webamp.org) for the other 90k.
- Drag-and-drop `.wsz` onto any window applies immediately; persist last-used skin path.

## 6. visionOS (Phase 3, design constraints now)

- Native Swift app: SwiftUI window for the player (a spatial reinterpretation of the Winamp layout, since bitmap skins won't map to volumetric UI), RealityKit `ImmersiveSpace` for the visualizer.
- Apple Music via native MusicKit: full playback. Visuals driven by tempo/key metadata + envelope estimation from `AVAudioEngine` where Apple permits taps on non-DRM content, otherwise procedural beat-sync. No loopback on visionOS; set expectations accordingly.
- Spotify on visionOS: remote control only (Connect API targeting another device), metadata visuals.
- Keep `PlayerState` and adapter contracts in a schema-shared form (JSON types) so the Swift app mirrors them.
- WebXR fallback: Butterchurn in Safari on visionOS driven by metadata; low priority.

## 7. Risks

| Risk | Severity | Mitigation |
|---|---|---|
| Widevine/ECS friction in Electron | High | Prove in Milestone 0; browser-Connect-device fallback ready |
| Spotify tightens dev mode further | High | Adapter isolation; Apple Music is the strategic backend; app degrades to Apple-only |
| SCK permission UX confuses users | Medium | Onboarding screen with one-click prompt + live meter confirming signal |
| Webamp internals resist adapter swap | Medium | Webamp exposes media hooks; worst case, fork (MIT) |
| MusicKit JS quirks in Electron | Medium | Known to work in Chromium; keep MusicKit in a dedicated webview if needed |
| Skin IP | Low | CC default skin, user-loaded everything else |

## 8. Milestones (Claude Code session plan)

- **M0 Spike (do first):** ECS Electron app plays one Spotify track via Web Playback SDK with user client ID, and one Apple Music track via MusicKit JS. No UI. This de-risks the entire project.
- **M1 Capture:** Swift SCK sidecar streaming PCM; renderer FFT; debug spectrum canvas moving to live audio.
- **M2 Player:** Webamp embedded, SpotifyAdapter wired to transport + marquee + playlist window.
- **M3 Apple parity:** AppleMusicAdapter to the same contract; source switcher.
- **M4 Vis:** classic vis window on real FFT (respecting viscolor.txt); Butterchurn detachable window, preset cycling, fullscreen.
- **M5 Ship v1:** settings/onboarding, skin drag-drop, safeStorage, error states, packaging + notarization, README setup guide.
- **M6+ (P1):** WebGPU scenes, Windows/WASAPI, media keys, beat-synced transitions.

## 9. Definition of done, v1

Fresh macOS user with Spotify Premium: installs Reamp, follows README to create a Spotify client ID, authorizes both services, grants capture permission, plays a playlist from the classic playlist window, watches bars move to the actual music, goes fullscreen Milkdrop, drags in a downloaded `.wsz`, and everything survives a 2-hour session. Under 15 minutes to first music.
